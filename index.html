01 · Features

Full Feature Set

Ten capabilities across core meeting intelligence and Arize-powered self-improvement. The first five are must-haves; the last five are the differentiators.

01 🎙️

Live Meeting Participant

What it does

The agent actually joins your meeting as a participant — it listens to everything being said and can speak back when needed. Think of it as a bot sitting in the call. It wakes up when someone says "Hey MeetAI" or when it detects it's being addressed.

How it works

We use Google Cloud Speech-to-Text Streaming API to transcribe audio in real time as people speak. When the agent needs to respond, Google Cloud Text-to-Speech converts the text to audio and plays it into the meeting channel. The bot joins via Recall.ai or a local WebRTC client.

What you need to build
  • Set up Recall.ai bot to join meeting calls
  • Wire Google Cloud STT streaming to capture live audio
  • Wire Google Cloud TTS to play agent responses back into the call
  • Implement wake-phrase detection ("Hey MeetAI")
Google STT Streaming · Google TTS · Recall.ai · WebRTC
02 🔍

Semantic Meeting Memory

What it does

The agent remembers everything from every past meeting. When someone asks "what did we decide about the pricing model last month?", it finds the exact moment in the exact meeting where that was discussed and gives you the answer with the date and who said it.

How it works

Every meeting transcript gets broken into small chunks (200 words each) and stored in Elasticsearch. Each chunk also gets converted into a vector embedding using Vertex AI. When a question comes in, we search using both keyword matching (BM25) and semantic similarity (vector search) at the same time — this hybrid approach finds the right answer even if the exact words don't match.

What you need to build
  • Set up Elasticsearch index with fields for meeting_id, date, speaker, text_chunk, and embedding
  • Build the indexing pipeline: transcript → chunk → embed → store
  • Build the hybrid search function that combines BM25 + vector results
  • Return results with meeting date and speaker name attached
Elasticsearch · Vertex AI text-embedding-004 · BM25 + vector hybrid search
03 📋

Agenda Drift Detector

What it does

Before the meeting starts, you give the agent the agenda. During the meeting, it keeps track of what's actually being discussed. If the conversation goes off-topic for more than about 90 seconds, the agent speaks up and says something like "We've been talking about team offsites for 2 minutes — that's not on today's agenda. Should we park it?"

How it works

The agent holds the agenda items as embeddings. Every 30 seconds, it takes the recent transcript and embeds it too, then measures how similar the current conversation is to each agenda item. If the similarity score drops below a threshold for long enough, it triggers the verbal intervention.

What you need to build
  • Embed each agenda item at meeting start
  • Run a background loop every 30 seconds that embeds the recent transcript window
  • Compare current topic embedding against agenda embeddings using cosine similarity
  • If drift score exceeds threshold for 90s, call the speak() tool with a redirect message
Gemini classification · cosine similarity · rolling 30s buffer · verbal intervention
04 💬

Real-Time Q&A

What it does

Anyone in the meeting can ask a question out loud — like "what was the budget we approved for Q3?" — and the agent will answer it in seconds, telling you exactly which meeting the answer came from and who said it.

How it works

The STT stream detects a question (ends with "?" or matches question patterns). The agent runs a hybrid search across all indexed meetings and documents. Gemini takes the top search results and generates a clear spoken answer. The agent then speaks it back using TTS.

What you need to build
  • Detect question intent from the live transcript stream
  • Call the search_meeting_history() tool with the question as the query
  • Pass retrieved chunks to Gemini to generate a grounded answer
  • Speak the answer with source attribution ("Based on the March 5th meeting...")
Hybrid search · Gemini · Google TTS · source attribution · <3s response
05 ✅

Live Action Item Tracker

What it does

The agent listens for phrases that signal someone is committing to do something — like "I'll handle that", "can you take care of the report?", or "let's make sure someone follows up on this". It automatically logs these as action items with who owns it, what the task is, and any deadline mentioned. These show up live on the dashboard.

How it works

Gemini classifies each transcript chunk for action item intent. When detected, it extracts the owner (who said it or who was assigned), the task description, and any deadline mentioned. This gets stored as a structured record and pushed to the dashboard in real time.

What you need to build
  • Build action item intent classifier using Gemini
  • Extract owner, task, and deadline from the matched text
  • Store structured action items in the database
  • Push updates to the React dashboard via WebSocket or polling
Gemini intent detection · structured extraction · real-time dashboard · WebSocket
06 📦

Post-Meeting Auto-Package

What it does

When the meeting ends, the agent automatically does three things: generates a clean summary of what was discussed and decided, indexes the full transcript so it's searchable in future meetings, and emails every attendee a report with the summary, action items, and a link to search the transcript.

How it works

Gemini summarises the full transcript into sections: decisions made, action items, key discussion points. The full transcript is chunked and indexed into Elasticsearch. The summary and action items are formatted into an email and sent via a mail API (SendGrid or similar).

What you need to build
  • Detect meeting end event (call disconnect or manual trigger)
  • Run Gemini summarisation prompt on the full transcript
  • Index the full transcript into Elasticsearch
  • Format and send the post-meeting email to all attendees
Gemini summarisation · Elasticsearch indexing · SendGrid email · meeting end detection
07 📊

Meeting Efficiency Score

What it does

After every meeting, the agent gives it a score out of 100 based on how well it went. The score is based on four things: how much time was spent on agenda topics (vs off-topic), how many clear decisions were made per hour, whether everyone participated or just one or two people dominated, and how clear the outcomes were. Over time you can see if your meetings are getting better.

How it works

Arize Phoenix runs LLM-as-a-Judge evaluations on the meeting transcript. Gemini acts as the judge and scores each dimension (0–1). The four scores are combined into a single efficiency score. This score is stored and displayed on the dashboard with a trend chart over time.

What you need to build
  • Define the four eval dimensions as Arize eval templates
  • Run Gemini-as-judge on the full transcript after meeting ends
  • Combine scores into a single efficiency metric
  • Store scores per meeting and render a trend chart on the dashboard
Arize LLM-as-a-Judge · Gemini judge · 4 eval dimensions · trend chart
08 🔭

Self-Introspection via Phoenix MCP

What it does

Every single decision the agent makes — every search it runs, every answer it gives, every drift detection — gets recorded as a trace in Arize Phoenix. The clever part is that the agent can then query its own past traces while it's running. Before answering a question, it can check "have I answered something like this before? How well did I do?" and use that to give a better answer this time.

How it works

OpenInference instrumentation automatically captures every tool call and LLM call as a trace and sends it to Phoenix Cloud. The Phoenix MCP server is configured as one of the agent's tools. At runtime, the agent can call phoenix.get_traces(), phoenix.get_evals(), and phoenix.query_spans() to look at its own history.

What you need to build
  • Install and configure OpenInference + GoogleADKInstrumentor
  • Set up Phoenix Cloud account and confirm traces are flowing
  • Configure Phoenix MCP server in the agent's tool config
  • Add a pre-answer check where the agent queries its own past performance for similar questions
OpenInference · GoogleADKInstrumentor · Phoenix MCP · phoenix.get_traces() · phoenix.get_evals()
09 ⚡

Auto-Prompt Optimisation

What it does

Once a week, the system looks at all the answers the agent gave that scored poorly (below 0.6 out of 1.0). It figures out what went wrong — was it retrieving the wrong sources? Were answers too vague? Did it miss drift? Then it automatically generates better versions of the agent's instructions, tests them against the failing examples, and if the new version is better, it replaces the old one. The agent literally rewrites its own instructions to get better.

How it works

A scheduled job queries Phoenix for low-scoring traces and groups them by failure type. Gemini generates improved system prompt variants for each failure cluster. The new prompts are evaluated against the failing examples using Arize evals. The prompt with the highest average score is promoted as the new active system prompt.

What you need to build
  • Build the weekly scheduled job (cron or Cloud Scheduler)
  • Query Phoenix API for traces with eval score < 0.6
  • Group failures by type using Gemini clustering, then generate prompt variants and run A/B eval
  • Store versioned prompts and promote the winner
Arize evals · Phoenix API · Gemini prompt generation · A/B evaluation · prompt versioning
10 🎤

Speaker Diarisation + Profiles

What it does

The agent knows who is speaking at all times. It doesn't just transcribe what was said — it tracks who said it. Over multiple meetings, it builds a lightweight profile for each person: what topics they usually own, how they communicate, what decisions they've been involved in. This lets the agent personalise its responses and know who to route a question to.

How it works

Google Cloud STT's speaker diarisation feature labels each segment of audio with a speaker ID. These IDs are matched to known attendees via the calendar/meeting invite. Over time, a speaker profile is built from their transcript history — topics they discuss, decisions they make, their typical speaking patterns.

What you need to build
  • Enable speaker diarisation in the Google STT config
  • Map speaker IDs to attendee names from the meeting invite
  • Store per-speaker transcript history in the database
  • Build a lightweight profile aggregation that tracks topics and decision involvement per speaker
Google STT diarisation · speaker ID mapping · profile aggregation · personalised responses

03 · Pipeline

Processing Pipeline

Every meeting interaction flows through six stages, each instrumented with OpenInference traces sent to Arize Phoenix.

1

Audio Ingestion & Transcription

Google Cloud Speech-to-Text Streaming API transcribes audio in real time. Speaker diarisation assigns speaker labels. A rolling 30-second buffer feeds the agent loop continuously.

Google STT Streaming · speaker diarisation · 30s buffer
2

Intent Classification

Each transcript chunk is classified: question, action item, decision, or general discussion. Agenda drift is scored by comparing the current topic embedding against agenda item embeddings.

Gemini classification · cosine similarity · agenda embeddings
3

Hybrid Search (Q&A path)

A hybrid Elasticsearch query combines BM25 keyword matching with Vertex AI vector similarity. Results are ranked by reciprocal rank fusion and returned with meeting date and speaker attribution.

Elasticsearch · BM25 + vector · RRF ranking · Vertex AI text-embedding-004
4

Response Generation & Speech

Gemini generates a grounded response using retrieved context. Google Cloud TTS converts it to speech and plays it in the meeting channel. Response is logged with source chunks for tracing.

Gemini · Google TTS · source attribution · OpenInference trace
5

Async Evaluation

After every response, an async Arize eval job scores it on relevance, groundedness, and helpfulness using Gemini as judge. Low-scoring traces (<0.6) are tagged for the weekly optimisation cycle.

Arize LLM-as-a-Judge · async eval · Gemini judge · trace tagging
6

Post-Meeting Indexing

Full transcript chunked into 200-token windows with 50-token overlap. Each chunk embedded with Vertex AI and indexed into Elasticsearch with meeting metadata, speaker labels, and extracted decisions.

200-token chunks · 50-token overlap · Vertex AI embedding · Elastic index
